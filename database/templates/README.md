# 📋 資料表設計規範 (Database Schema Specification)

## 1. 分表策略 (Sharding by Company)

* **核心邏輯**：為避免單一資料表因資料量過大導致索引失效，採用「**一家公司一張實體表**」設計。此外，為精簡結構，**上市公司（L）與公發公司（X）共用同一個行業別實體表**（不再區分 `_l_` 或 `_x_` 檔案），並透過新增的 `market_type` 欄位來標記資料來源身分。
* **動態表名**：表名後綴為佔位符 `{COMPANY_CODE}`（例如：`t187ap07_ci_2330`），由後端程式執行時動態替換。
* **市場註記欄位**：資料表內含 `market_type` 屬性欄位（`L` 代表上市，`X` 代表公發），**此欄位不納入主鍵**。當公司身分發生變更時（如公發轉上市），透過 `ON DUPLICATE KEY UPDATE` 機制自動更新該欄位，維持資料最新狀態。
* **複合主鍵**：主鍵統一設定為 `PRIMARY KEY (year, quarter)`，確保單一公司在同年度同季別中只有唯一一筆財報紀錄，從資料庫底層杜絕因重複上架或 ETL 異常產生的髒資料與重複紀錄。


## 2. 欄位過濾與保留原則

* ❌ **移除「出表日期」(`date_issued`)**：動態財報著重於會計期間資料，出表日期僅作參考，故予以移除。
* ⭕ **保留「公司代號」與「公司名稱」**：為確保每筆資料列皆具備「自解釋性」，並便於未來進行跨公司的大盤或產業別合併分析（`UNION ALL`），此二欄位予以保留。
* ⭕ **嚴格保留原始科目順序**：除上述調整外，所有會計科目完全依據政府開放資料之原始 JSON 欄位相對順序排列，確保完美的資料對齊。

## 3. 資料型態優化

政府開放資料之原始格式一律為 `string`，本專案在實體表中將其優化為正確的資料型態：

* **年度、季別**：採用 `VARCHAR`。
* **金額、股數科目**：統一轉為 `DECIMAL(20, 2) DEFAULT 0.00`，以利進行數值計算與加總。
* **每股參考淨值**：統一轉為 `DECIMAL(20, 2) DEFAULT 0.00`，避免浮點數計算誤差。

## 4. 系統稽核欄位

* 尾端統一新增 `updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`。
* **效益**：首次寫入自動記錄時間；若遇財報更正重編（透過 `ON DUPLICATE KEY UPDATE` 覆蓋）時，資料庫會自動刷新時間，完美追蹤數據時效性。
