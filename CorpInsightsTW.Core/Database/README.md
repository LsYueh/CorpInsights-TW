# 📋 資料表設計規範 (Database Schema Specification)

## 1. 資料儲存與索引策略 (Storage & Indexing Strategy)

* **核心邏輯**：放棄動態拆表（Sharding），改採「**行業別大寬表**」設計。全台灣不論上市、上櫃或公發公司，只要屬於同一個行業別，其財報數據皆集中儲存於單一實體表中（例如：所有一般業資產負債表皆存於 `t187ap07_ci`）。
* **市場身分合併**：上市公司（L）與公發公司（X）共用同一個行業別實體表，不區分檔案。表內統一配置 `market_type` 屬性欄位（`L`: 上市, `X`: 公發）以追蹤數據來源，當公司身分轉換時，透過 ETL 覆寫機制維持最新身分。
* **複合主鍵（Composite Primary Key）**：
統一設定為 **`PRIMARY KEY (company_code, year, quarter)`**。
從資料庫底層強制實施業務約束，確保「**單一企業在特定年度與季度中，僅能存在唯一一筆財報紀錄**」，徹底杜絕重覆上架、ETL 併發異常所導致的重複數據。
* **次級索引優化（Secondary Index）**：
針對跨公司排名與年度橫向分析情境，額外建立複合索引 **`INDEX idx_year_quarter (year, quarter)`**，確保在執行特定季別的選股條件或財務指標排名時，查詢效能同樣達到毫秒級響應。

## 2. 欄位過濾與保留原則

* ❌ **移除「出表日期」(`date_issued`)**：動態財報著重於特定會計期間（年度與季別）的數據實質，出表日期僅屬系統生成時間，對財務分析無實質意義，故予以移除。
* ⭕ **增設「市場註記」(`market_type`)**：為配合上市/公發通用大寬表架構，於核心識別區段新增 `market_type` 欄位（儲存 `'L'` 代表上市，`'X'` 代表公發），用以追蹤與識別資料的初始 API 來源身分。
* ⭕ **保留「公司代號」與「公司名稱」**：為確保大寬表中每一列（Row）數據皆具備完整的「自解釋性」，無須透過複雜的 `JOIN` 即可直接識別企業實體，並做為複合主鍵（`PRIMARY KEY`）的核心前導欄位。
* ⭕ **嚴格保留原始科目順序**：除上述核心識別欄位（年度、季別、公司代號、公司名稱、市場註記）統一置於表頭外，其餘所有會計科目完全依據政府開放資料之原始 JSON 欄位相對順序排列，確保完美的資料盲刷（Blind Bulk Insert）與對齊。

## 3. 資料型態優化

政府開放資料之原始格式一律為 `string`，本專案在實體表中將其優化為正確的資料型態：

* **年度、季別**：採用 `VARCHAR`。
* **金額、股數科目**：統一轉為 `DECIMAL(20, 2) DEFAULT 0.00`，以利進行數值計算與加總。
* **每股參考淨值**：統一轉為 `DECIMAL(20, 2) DEFAULT 0.00`，避免浮點數計算誤差。

## 4. 系統稽核欄位

* 尾端統一新增 `updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`。
* **效益**：首次寫入自動記錄時間；若遇財報更正重編（透過 `ON DUPLICATE KEY UPDATE` 覆蓋）時，資料庫會自動刷新時間，完美追蹤數據時效性。
